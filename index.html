Our Planning

  • Define Use cases
  • Estimate each story as an epic
  • Rough Milestones: Campaign-Ready, Feature-Parity, "New Stuff"
Describe difference between user story and use case. The story "user wants to donate" is implemented in 3-4 use cases: Show donation form, add donation, display donation result, Paypal notification

Our working mode

  • Work on each story, breaking it down into smaller tasks as needed
  • Tasks are picked from the board
  • No sprints, one big burndown chart

Learning

Progress tracking based on rough story estimates is highly unreliable.

Learning

Use cases describe functionality and neglect "infrastructure" effort

Keep infrastructure concerns in mind when breaking down tasks and create separate tickets for them

Lesson: "Distributed" teams can work well and can self-organize/priotitize

"Distributed" means not working in the same place or at the same time

Encourage shared commitment.

Work on big tasks (e.g. use cases or other big todos) with more than one person.

Advantages

  • knowledge sharing
  • improve and increase interpersonal communication
  • have a better sense of achievement and progress (by not working in parallalel)

Don't be a lone explorer

When trying out "new stuff" (eg. new patterns, libraries or languages), learn together.

That means to do pair programming, intensive reviews and demos.

Review-First

Before working on your code, always check if code needs to be reviewed!

Use code for reviews

Make followup-PRs instead of commenting.

More lines, less cleverness

What is better, this

foreach ( ['name', 'street', 'zip', 'city', 'email' ] as $field ) {
	$setter = 'set' . ucfirst( $field )
	$address->$setter( $_REQUEST[$field] );
}

or this

$address->setName( $_REQUEST['name'] );
$address->setStreet( $_REQUEST['street'] );
$address->setZip( $_REQUEST['zip'] );
$address->setCity( $_REQUEST['city'] );
$address->setEmail( $_REQUEST['email'] );
The latter is better because it eases refactoring (renaming of settings in the address object) and decouples HTML forms fields from entity field names.

Continous integration/deployment

For every pull request, do the following checks

  • Coding style
  • All the tests (Unit/Integration/System, etc)
  • Code metrics

Check both PHP and JavaScript code!

Master is always clean and deployable.
