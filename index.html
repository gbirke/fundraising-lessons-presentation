Learnings from the development of the the new fundraising frontend application

What is this about?

  • Organization Perspective
  • Coding Perspective

Our working mode

  • Define User stories/Use cases
  • Estimate each story as an epic
  • Work on each story, breaking down into smaller stories as needed
  • Tasks are picked from the big board
  • No sprints, one big burndown chart

Problem

Burndown chart based on rough story estimates is highly unreliable

Solution

  • Distribute stories to milestones (2-4 weeks)
  • Break down stories into tasks for current milestone
  • Separate burndown chart(s) for each milestone

Milestones also help with celebrating/appreciating progress

Mention that we had three "Milestones" (campaign-ready, feature parity and "new stuff") - way too broad.

Problem

Use cases describe functionality and neglect "infrastructure" effort

Solution

Keep infrastructure concerns in mind when creating tasks.

  • Database
  • Web request handling (routing)
  • Validation
  • Internationalization and Localization
  • External services
  • Logging

Communication

  • Task board (who does what)
  • Code reviews
  • Sporadic dailies
  • In-Person for planning and WTF moments during code review

More lines, less cleverness

What is better, this

foreach ( ['name', 'street', 'zip', 'city', 'email' ] as $field ) {
	$setter = 'set' . ucfirst( $field )
	$address->$setter( $_REQUEST[$field] );
}

or this

$address->setName( $_REQUEST['name'] );
$address->setStreet( $_REQUEST['street'] );
$address->setZip( $_REQUEST['zip'] );
$address->setCity( $_REQUEST['city'] );
$address->setEmail( $_REQUEST['email'] );
The latter is better because it eases refactoring (renaming of settings in the address object) and decouples HTML forms fields from entity field names.
